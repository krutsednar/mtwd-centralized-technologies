# Architecture Audit — mct-staging
**Date:** 2026-05-23
**Stack:** Laravel 12 · PHP 8.3 · Filament 3 · Livewire 3 · Tailwind 4 · PostgreSQL (mct_proddb schema) · Pest 3
**Scope:** Read-only audit of `app/`, `database/migrations/`, `routes/`, `config/`, `tests/`, `resources/views/`
**Total LOC examined:** ~2,936 (models) + ~6,444 (Filament resources) + ~1,463 (Livewire) + 80 migrations

### 2.2 Cross-Domain Coupling (Leaks)

These imports cross domain boundaries today; the future plan must keep them or rewrap them:

- `app/Filament/Gsms/Resources/DriverResource.php:8`, `EquipmentResource.php:8`, `HeavyEquipmentResource.php` (via Pages), `VehicleResource.php` (via Pages), `TripTicketResource.php:9` — all GSMS resources import `App\Models\Profile` (HRIS). Justified for "assigned to employee" lookups, but should ideally go through a `ProfileLookup` query service or a read-model.
- `app/Filament/Gsms/Resources/EquipmentResource.php:10`, `VehicleResource.php:10`, `HeavyEquipmentResource.php:9`, `Pages/ListVehicles.php:8`, `Pages/ListHeavyEquipment.php:8`, `Hris/Resources/AttendanceResource.php:6` — all import `App\Models\Division` (OrgStructure).
- `app/Providers/Filament/HrisPanelProvider.php:41` and `AdminPanelProvider.php:46` and `GsmsPanelProvider.php:47` all register `\App\Filament\Home\Resources\HomeResource\Widgets\CustomAccountWidget::class` — a **Home panel widget being referenced across three other panels**. Should live in `App\Shared\Filament\Widgets`.
- `app/Filament/Hris/Resources/LeaveApplicationResource.php:9` imports `App\Models\ServiceRecord` purely to autofill `position` + `salary` on profile select (line 87-92). This is HRIS-internal, fine.
- `app/Filament/Hris/Resources/AttendanceResource.php:6` imports `Division` for the bulk-print dropdown — Attendance → OrgStructure leak.

**Universal model issues:**
- 39 of 41 models have **no factory**. Only `User` and `Attendance` have factories.
- Casts mixed between `$casts` property (older style) and `casts()` method (Laravel 12 preferred). Standardise on `casts()`.
- Deprecated `$dates` property used on `Profile`, `User`, `Attendance`, `ServiceRecord` — Laravel 12 wants `casts()`.
- Many models miss return type hints on relationships. Examples: `Profile::division()`, `Profile::tripTickets()`, `User::division()`, `ServiceRecord::profile()`.
- No models extend a shared base class — useful for `LogsActivity`, soft-deletes, observer wiring.

## 5. Risks & Trade-offs

### What can break
| Risk | Mitigation |
|---|---|
| **Filament resource discovery breaks** when resources move under `app/Domain/.../Filament/Resources` — current `discoverResources()` calls all point at `app_path('Filament/<Panel>/Resources')`. | Add a second `discoverResources()` call per panel pointing at the domain path, e.g. `->discoverResources(in: app_path('Domain/FaceBiometrics/Filament/Resources'), for: 'App\\Domain\\FaceBiometrics\\Filament\\Resources')`. Keep both call sites until migration finishes. |
| **Livewire component resolution fails** — `config/livewire.php` `class_namespace = 'App\\Livewire'` won't find domain Livewire components. | Either keep all Livewire under `App\\Livewire` (give up on per-domain Livewire) or use `Livewire::namespace('App\\Domain\\FaceBiometrics\\Livewire', 'face-biometrics')` in `FaceBiometricsServiceProvider`. Earlier in this session, Filament's `FaceBiometricEnrollmentStats` widget had to be explicitly added to `widgets([...])` for the same reason — Livewire only auto-discovers within configured namespaces. |
| **Filament Shield permission strings break** — Shield generates permission names from resource namespaces. Moving namespaces invalidates existing permission rows. | Run `php artisan shield:generate --resource=...` after each move, write a one-shot migration to remap old permission names to new. |
| **Spatie Activitylog `log_name`** is derived from class FQCN; the activity_log table will show old class names for historical entries — readable but confusing. | Accept the cosmetic blip; add an `auto-formatter` view-side if needed. |
| **`config:cache` in production** — `OnlineHrisService::writeTokenToEnv` already breaks this. Once domains add their own service providers with closures (`bind(fn () => ...)`), `config:cache` won't break but `route:cache` may if controllers are closures. | All controllers should be class-based after Phase 1; avoid closures in domain providers. |
| **Tests don't exist yet** — refactoring with zero tests is risky. | Phase 2's pilot includes 5 first tests; expand to ≥1 happy-path test per moved domain before declaring the move "done". |
| **Filament Shield's policies** are wired to the existing model FQCN; moving `App\Models\Profile` → `App\Domain\Hris\Models\Profile` requires updating every policy class' `$model` reference + every resource's `getModel()`. | Mechanical sed; verify by running `php artisan shield:generate` against the new namespace. |
| **PostgreSQL schema name `mct_proddb`** is hardcoded in `FaceEmbedding.php:42,58,65,90` `DB::statement('SET search_path TO mct_proddb, public')`. Multi-environment (staging vs prod) breaks if schema name changes. | Read from `config('database.connections.pgsql.search_path')` instead of hardcoding. |

### Tooling / IDE updates needed
- PHPStorm: refresh "Laravel" / "Filament" IDE helper auto-completes after each domain move.
- Pint config: no path adjustments needed — `vendor/bin/pint` walks all PHP files.
- Composer `autoload.psr-4`: no change needed if domain paths follow `App\Domain\X` → `app/Domain/X/`.
- `bootstrap/providers.php` — add each domain service provider.
- Filament Shield permission cache: clear after every batch.

### Things that become harder
- **Cross-domain queries** (e.g. "all profiles in division X with active face enrollment") will need explicit query objects or read-models. Currently it's a free-for-all of `App\Models\*` imports.
- **Code search for users new to the project** — moving from a flat `app/Models/Profile` to a nested `app/Domain/Hris/Models/Profile` adds a click. Counter-acted by clear domain naming.
- **Refactor PRs grow** during migration. Recommend one domain per PR.

## 6. Open Questions (decisions required before any code moves)

1. **Section under Section, or only under Division?** The `Division::ALLOWED_PARENT_TYPES` entry just added lists `[TYPE_DIVISION]` for `TYPE_SECTION`. Should sections be nestable (Section under Section)? If yes, the array should be `[TYPE_DIVISION, TYPE_SECTION]`.
2. **Retire `BiometricEnrollment` (v1) entirely?** It still has a model, table, Filament resource, Livewire components, and Jobs reference it. If Face Biometrics v2 is the future, set a deprecation deadline and remove. If both must coexist, document why.
3. **Where does `Profile` live?** It's the read-model for HRIS but is referenced by GSMS (driver assignments), Attendance (FK), FaceBiometrics (FK), Leave (FK). Options:
   a. `App\Domain\Hris\Models\Profile` and let everyone import it (current state, just relocated).
   b. `App\Shared\Models\Profile` (treat it as cross-cutting identity).
   c. Split into `App\Shared\Models\Employee` (identity, FK target) and `App\Domain\Hris\Models\ProfileDetails` (PDS-heavy fields).
4. **`User` vs `Profile`** — these are two parallel identity tables. Should they merge? Today `User` is the auth row and `Profile` is the HRIS employee row, linked by `employee_number` string match. Brittle.
5. **Convert `divisions.type` to varchar (cheap) or PostgreSQL ENUM type (stricter)?** Plain enum column blocks future values; adding `section` once required a migration. Recommend varchar + app-level Backed Enum validation in the model + a CHECK constraint listing allowed values (easier to ALTER than a typed enum).
6. **Will the project gain a real REST/JSON API soon?** If yes, scaffold `app/Http/Resources/` and `app/Http/Requests/` in Phase 1. If not, defer.
7. **Are we targeting Pest 3 only, or PHPUnit-style tests too?** Boost guidelines say Pest. Confirm.
8. **Activity-log retention** — the table grows unbounded. Configure a `prune` schedule? Affects which domain "owns" audit (Shared vs each domain).
9. **Should `App\Livewire\` stay flat or also be split per domain?** Three approaches:
   a. Keep `App\Livewire\<Domain>\*` as today and just rename `Hris/` → `Domain/Hris/Livewire/`.
   b. Move all Livewire under `App\Domain\X\Livewire\` and register each via service provider.
   c. Keep Livewire flat under `App\Livewire\` permanently as a UI-only layer (lowest risk).
10. **Who is responsible for `OrgStructure` change validation?** Currently spread across `Division::canBeChildOf()` (model), `StructureManagementResource\Pages\ListOrgStructure::reparent()` (page), and `SupervisorManagementResource` (resource). Consolidate into `App\Domain\OrgStructure\Actions\*` so there is one place to test.
